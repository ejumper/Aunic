## tool_id
For model-generated tool calls, the `tool_id` is generated by the model's API response. For Anthropic it's the `id` field on `tool_use` content blocks, for OpenAI it's `tool_calls[].id`. Aunic stores the API-provided ID in the transcript and echoes it back in the corresponding tool result.

For user-initiated `@web` search and fetch rows, Aunic synthesizes `tool_id` values in the form `user_<n>` so those transcript pairs behave the same way as model-generated tool rows.

## In-memory message list vs. transcript table
The model server is stateless — every API call requires the full message history. During a run, Aunic maintains an **in-memory message list**, aka `run-log` that starts as the translated transcript + note-snapshot + user-prompt, then grows turn-by-turn as tool calls and results are appended. When the run ends, this list is discarded.
- **Persistent tools** (web_search, web_fetch, work-mode tools): tool_call/tool_result rows are added to both the in-memory list and the transcript table.
- **Ephemeral tools** (note-edit, note-write): tool_call/tool_result are added to the in-memory list only. They are not stored in the transcript. The note-snapshot reflects the cumulative result of those edits, so future runs see the outcome but not the edit history.
- **Thinking blocks**: added to the in-memory list (the API requires them for subsequent turns in the same run) but not stored in the transcript. See below.

## Thinking blocks
Anthropic's API can return `thinking` content blocks (extended thinking) alongside text and tool_use blocks. These are large, ephemeral, and not useful as historical context. **Thinking blocks are excluded from the transcript** — they are not stored as rows. They exist only during the run that produced them and are discarded after.

## Adding rows to the `transcript`
Aunic (the program) writes transcript rows — the model never touches the markdown file directly. No model tokens are spent on transcript bookkeeping.

### Write mechanic
Rows are written by reading the full active markdown file, appending the new row after the last table row (or after the `---\n# Transcript` header and column headers if the table is empty), and writing the full file back.

### Row numbering
The `#` column is auto-incremented from the last row's number. If rows are deleted and gaps appear (1, 2, 5, 6), they are not renumbered — the parser relies on row order, not the `#` value.

### When rows are written
- **User `message` row**: written when the user hits send.
- **Assistant `message` row**: written when the API response arrives.
- **`tool_call` row**: written immediately when the API response arrives, before the tool executes. This ensures the transcript reflects what the model asked to do even if execution fails or Aunic crashes mid-run.
- **`tool_result` or `tool_error` row**: written when tool execution completes.
- **Mixed assistant turns** (text + tool calls in one API response): each content block is written as a separate row, in the order it appears in the API response's content array (typically text first, then tool_use blocks). The grouping back into a single assistant message happens at translation time (see `notes-and-plans/building-context/transcript-to-api.md`, Step 2).

### What goes in the `content` cell
The `content` cell stores the arguments (for calls) and the return value (for results) as JSON. The principle is: tool_call content = the arguments object exactly as the API returned it; tool_result content = whatever Aunic sent back to the model as the tool result, JSON-encoded.

| tool | tool_call content | tool_result content |
|------|------------------|-------------------|
| `web_search` | `{"queries":["search query"]}` | `[{"url":"...","title":"...","snippet":"..."}]` |
| `web_fetch` | `{"url":"https://example.com"}` | `{"url":"...","title":"...","snippet":"..."}` (compact summary — full page is in-memory only, see fetch-tool.md) |
| `edit` | `{"file_path":"src/main.py","old_string":"old text","new_string":"new text"}` | `"Edit applied to src/main.py"` or error message |
| `write` | `{"file_path":"src/new.py","content":"file contents..."}` | `"File written to src/new.py"` |
| `bash` | `{"command":"ls -la"}` | `"drwxr-xr-x 5 user ..."` (stdout/stderr) |
| `read` | `{"file_path":"src/main.py","offset":10,"limit":20}` | `"line 10 content\nline 11 content\n..."` |
| `grep` | `{"pattern":"TODO","glob":"*.py"}` | `[{"file":"src/main.py","line":14,"text":"# TODO fix this"}]` |
| `glob` | `{"pattern":"src/**/*.ts"}` | `["src/index.ts","src/utils/helpers.ts"]` |
| `note_edit` | `{"old_string":"old text","new_string":"new text"}` | in-memory only structured patch result |
| `note_write` | `{"content":"full note body"}` | in-memory only structured patch result |

For tools with large outputs (`read`, `bash`, `edit`, `write`), the transcript may store a bounded structured payload while the in-memory run-log keeps the richer provider-facing result. The transcript remains the durable record; the run-log remains the richer per-run context.

### Deleting rows from the `transcript`
Tool entries in the transcript are always paired: a `tool_call` row and a `tool_result` row sharing the same `tool_id`. In the Aunic UI, only `tool_result` rows are rendered (the `tool_call` rows are hidden). When the user deletes a rendered row:
- **Tool entries**: deleting the visible `tool_result` row also deletes the matching `tool_call` row (matched by `tool_id`). This prevents orphaned rows that would break API translation.
- **Chat messages** (`type: message`): deletion removes just that single row.
- Row numbers are not renumbered after deletion — gaps are left in the `#` column.
